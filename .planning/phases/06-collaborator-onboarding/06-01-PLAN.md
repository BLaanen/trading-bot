# Plan: Collaborator Onboarding Experience

## Task 1: Add user_config.json support to config.py

**File:** `config.py`

Add loading logic to `AgentConfig.__post_init__`:

1. At the top of `__post_init__`, before the paper exploration mode block:
   - Resolve path via `pathlib.Path(__file__).parent / "user_config.json"` (NOT `os.getcwd()` — must work when config.py is imported from other directories)
   - Try to read the file. Catch only `FileNotFoundError` and `json.JSONDecodeError` — do not catch broad `Exception` (let permission errors surface)
   - If it exists and contains `starting_capital`, override `self.starting_capital` with that value
   - Derive the target multiplier from defaults: `ratio = 25_000 / 10_000` (2.5x). Override `self.target_capital = self.starting_capital * ratio`. Do not hardcode 2.5.
   - Guard: if `self.target_capital <= self.starting_capital`, set `self.target_capital = self.starting_capital * ratio` (handles edge case where user enters $25,000 which equals the default target)
   - If the file doesn't exist or is malformed, silently use defaults
2. Keep the existing defaults unchanged as fallbacks
3. Execution order in `__post_init__`: (1) load user_config.json overrides, (2) apply paper_exploration_mode adjustments. Capital fields don't interact with exploration mode but this order is clearest.

**Done when:**
- `AgentConfig()` with no user_config.json returns starting_capital=10000 (unchanged)
- `AgentConfig()` with `{"starting_capital": 5000}` in user_config.json returns starting_capital=5000 and target_capital=12500
- `AgentConfig()` with `{"starting_capital": 25000}` still has target_capital > starting_capital
- Malformed JSON doesn't crash — falls back to defaults
- `FileNotFoundError` is silent, `PermissionError` is not caught

## Task 2: Add starting capital selection and paper trading education to setup.sh

**File:** `setup.sh`

Add a new step between the current Step 3 (Alpaca keys) and Step 4 (connection verification). This becomes the new Step 4, bumping the remaining steps to 5 and 6. **Update all existing step headers from "X of 5" to "X of 6".**

**Re-run detection:** Before prompting, check if `user_config.json` already exists. If so, read the current `starting_capital` and show: "You already have a starting amount set: $X,XXX. Press Enter to keep it, or type a new amount:" This prevents silent overwrite on re-run.

**New Step 4: Choose your starting amount**

```
Step 4 of 6: Choose your starting amount

  Alpaca gives you $100,000 in paper money, but that's not
  realistic for learning — most people don't start with $100K.

  This system tracks a separate budget that controls how much
  it actually uses. Position sizes, risk limits, and portfolio
  tracking all work from YOUR number, not Alpaca's.

  Common starting amounts:
    $1,000  — very conservative, 1-2 positions at a time
    $5,000  — room for 3-4 positions with proper sizing
    $10,000 — the default, good balance of diversification and learning
    $25,000 — avoids pattern day trader rules if you go live later

  Enter your starting amount in dollars (or press Enter for $10,000):
```

After the user enters a number:
1. **Use a `while true; do ... break; done` loop** for input validation (required because `set -euo pipefail` would exit the whole script on a validation failure otherwise)
2. Validate it's a positive number >= 500. On invalid input, show: "Please enter a number of 500 or more. Below $500, the 2% risk rule produces positions too small to trade." Then re-prompt.
3. Write to `user_config.json` using `$PYTHON -c "import json; ..."` — wrap in error handler that prints a clear message on failure rather than letting bash exit
4. Print confirmation: "✓ Starting with $X,XXX — the system will size positions and track risk based on this amount."

**Done when:**
- setup.sh asks for starting amount and writes user_config.json
- Default (Enter) writes 10000
- Custom amount like 5000 writes 5000
- Invalid input (letters, negative, <500) shows a helpful message and re-prompts
- Re-running setup.sh with existing user_config.json shows current value and offers to keep it
- All step headers say "X of 6"

## Task 3: Add market scope and collaborator guide to README

**File:** `README.md`

Add two new sections:

**Existing README fix:** In the "How risk management works" section, change the hardcoded "$10,000 account, that's $200 per trade" example to use percentages only (e.g., "2% of your portfolio"). Dollar amounts tied to $10K will be wrong for users who pick a different starting capital.

**Section: "What markets does this cover?"** (after "How risk management works")

Content should explain:
- This system trades US stocks and ETFs through Alpaca
- Alpaca also offers ~700 ADRs (American Depositary Receipts) — shares of international companies that trade on US exchanges during US hours. So you can trade companies like Toyota or Nestlé, but through their US-listed ADR, not directly on the Tokyo or Swiss exchange.
- No European exchanges, no forex, no futures
- Crypto is available on Alpaca but not wired into this system
- Paper trading works from any country — no geographic restrictions. Live trading from Europe requires verification with Alpaca.
- All scanning and trading happens during US market hours: 9:30 AM to 4:00 PM Eastern Time, Monday through Friday

**Section: "For Collaborators" (using Claude Code)** (after "Project structure")

This section is for people using Claude Code in this repo. Content:

1. **Starting a session** — when you open Claude Code in this repo, Claude reads CLAUDE.md automatically and checks the system status (open positions, today's P&L, scheduler health). You don't need to tell it anything — it starts by showing you what's going on.

2. **Useful commands** — list the slash commands with one-line descriptions:
   - `/status` — what's open, today's P&L, system health
   - `/edge-check` — which strategies are earning vs fading
   - `/run-scan` — run the scanner manually right now
   - `/weekly-brief` — Friday-style summary of the week
   - `/what-happened-today` — narrative of today's activity
   - `/why-this-trade` — explain a specific trade
   - `/pause-trading` / `/resume-trading` — stop/start the automated schedule

3. **When to clear a session** — Claude Code sessions accumulate context as you work. If a session gets long or you're switching topics, type `/clear` to start fresh. Claude will re-read the system state files automatically, so nothing is lost — the ground truth lives in positions.json, trades.csv, and the log files, not in conversation memory.

4. **What persists across sessions** — positions, trade history, portfolio value, edge tracking, and configuration all live in files on disk. Clearing a session or starting a new one doesn't affect any of this. The only thing lost on clear is the conversation itself.

**Done when:**
- README has a clear market scope section
- README has a collaborator guide section with slash commands
- Both sections are written for someone who doesn't know trading vocabulary
