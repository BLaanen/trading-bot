# Plan: Scanner Cache, Universe Prebuild & File Permissions

Covers ROADMAP milestones 2 (N+1 scanner), 5 (file permissions), and 6 (universe prebuild).

## Task 1: Pass pre-initialized provider to scanner

**Files:** `scanner.py`, `orchestrator.py`

**Current problem:** `scanner.py` has a module-level `_provider = None` (line 58) and `_get_provider()` (line 62) that lazily initializes on first call. This works fine since it's memoized — but the provider is initialized inside the scanner module with no control from the orchestrator. If the orchestrator runs scan twice (or scanner is imported from multiple paths), the global state is fragile.

**Changes to scanner.py:**
1. Add `provider=None` parameter to `run_full_scan(config, provider=None)` (currently just takes config)
2. If provider is passed, use it instead of calling `_get_provider()`
3. Add `provider=None` parameter to `fetch_data(ticker, period="1y", provider=None)`
4. `fetch_data` uses passed provider if available, falls back to `_get_provider()` if None (backward compat)
5. Add `provider=None` parameter to all 5 strategy functions: `scan_pullback` (line 182), `scan_consolidation_breakout` (line 256), `scan_ma_bounce` (line 331), `scan_sector_momentum` (line 403), `scan_powerx` (line 484)
6. Each strategy function passes `provider` to its `fetch_data(ticker, provider=provider)` call
7. `run_full_scan` passes `provider` when calling each strategy function
8. Keep `_provider` global and `_get_provider()` for backward compatibility with direct `--scan` calls

**Changes to orchestrator.py:**
1. In `run_full_pipeline`, before `step_scan` (line ~348), initialize provider:
   ```python
   from data_provider import get_provider
   provider = get_provider()
   ```
2. Pass to `step_scan(config, provider=provider)`
3. Update `step_scan` to accept and forward `provider` kwarg to `run_full_scan`

**Done when:**
- `run_full_pipeline` initializes provider once, passes it to scanner
- Direct `python orchestrator.py --scan` still works (uses lazy init)
- All existing tests pass

## Task 2: Pre-build universe cache on startup

**Files:** `universe.py`, `orchestrator.py`, `at_open.sh`

**Current behavior:** `universe.py` line 285-292 builds and caches to `.universe_cache.json`. The `build_universe()` function (line ~230) scrapes Wikipedia if cache is stale (7-day TTL per line ~245). This blocks the scan step.

**Changes to universe.py:**
1. Add a new top-level function:
   ```python
   def ensure_cache() -> dict | None:
       """Ensure universe cache exists and is fresh. Returns cached data or None."""
   ```
2. Implementation:
   - Check if CACHE_FILE exists and `mtime` is within `REFRESH_DAYS` (reuse the existing 7-day TTL constant, not a separate 24h value)
   - If fresh: load and return it (fast path, no network)
   - If stale or missing: try `build_universe()` with a 30-second signal alarm timeout
   - If build succeeds: return new data
   - If build times out or fails: if stale cache exists, return stale cache with `logging.warning()`; otherwise return None (scanner will use watchlist only)
3. Use `signal.alarm(30)` for the timeout on macOS:
   - Save pre-existing SIGALRM handler via `old_handler = signal.signal(signal.SIGALRM, timeout_handler)` before setting alarm
   - Restore via `signal.signal(signal.SIGALRM, old_handler)` in finally block
   - Cancel alarm with `signal.alarm(0)` in finally block
   - Document constraint: must run on main thread, before any threaded network activity

**Changes to orchestrator.py:**
1. In `run_full_pipeline`, add before step_scan (line ~347):
   ```python
   from universe import ensure_cache
   ensure_cache(max_age_hours=24)
   ```
2. This is a fire-and-forget call — scanner will use whatever cache exists

**Changes to at_open.sh:**
1. No changes needed — at_open.py calls orchestrator which now handles cache

**Done when:**
- Fresh cache (< 24h): no Wikipedia request, pipeline continues immediately
- Stale cache: rebuilds, but times out after 30s and uses stale version
- No cache at all: builds or falls back to FALLBACK_TICKERS
- `python orchestrator.py --scan` also benefits (universe.py's existing lazy cache still works)

## Task 3: Set state file permissions to 0o600

**Files:** `risk_manager.py`, `executor.py`, `trade_tracker.py`, `edge_tracker.py`, `universe.py`

Add `os.chmod(filepath, 0o600)` after every write to state files. These files contain trading data and should be owner-readable only.

**Specific write locations:**

1. **risk_manager.py line 160** — `save_positions` writes `positions.json`
   After `json.dump(...)` and file close, add: `os.chmod(POSITIONS_FILE, 0o600)`

2. **executor.py line 498** — `_log_order` writes `order_log.json`
   After write, add: `os.chmod(ORDER_LOG, 0o600)`

3. **trade_tracker.py line 30** — `_ensure_file` writes `trades.csv`
   After write, add: `os.chmod(filepath, 0o600)`

4. **trade_tracker.py line 78** — append path `open(TRADES_FILE, "a")`
   After close, add: `os.chmod(TRADES_FILE, 0o600)` (covers first-create-via-append case)

5. **edge_tracker.py line 135** — `_save` writes `edge_tracker.json`
   After write, add: `os.chmod(EDGE_FILE, 0o600)`

6. **universe.py line 291** — cache write to `.universe_cache.json`
   After write, add: `os.chmod(CACHE_FILE, 0o600)`

7. **orchestrator.py line 67** — `save_last_run` writes `last_run.json`
   After write, add: `os.chmod(LAST_RUN_FILE, 0o600)`

8. **at_open.py lines 294, 349** — `last_run_file.write_text(...)` direct writes
   After each `write_text()`, add: `os.chmod(last_run_file, 0o600)`

**Implementation pattern (same everywhere):**
```python
with open(filepath, "w") as f:
    json.dump(data, f, indent=2)
os.chmod(filepath, 0o600)
```

Ensure `import os` is present at top of each file (most already have it).

**Done when:**
- All 6 write locations have chmod after write
- New state files created with 0o600 permissions
- Existing state files get corrected on next write
- All tests pass (tests may need tmpfile permissions — check test_executor.py and test_simulation.py)
