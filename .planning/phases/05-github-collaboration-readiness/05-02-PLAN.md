# Plan: README, Documentation & Contributing Guide

Covers ROADMAP milestones 1 (GitHub repo), 2 (README), 5 (PowerX docs), and 6 (contributing guide).

## Task 1: Write README.md

Create `README.md` at repo root. Structure:

**Section 1: Title & one-liner**
```
# Trading Bot
Algorithmic paper trading system for Alpaca, scanning 5 strategies across 300+ stocks.
```

**Section 2: What it does (bullet list, 6-8 items)**
- Scans S&P 500 + NASDAQ-100 for setups across 5 strategies (Pullback, Breakout, MA Bounce, Sector Momentum, PowerX)
- Places bracket orders at Alpaca (entry + stop-loss + take-profit in one atomic order)
- Sizes positions using the 2% risk rule — never risks more than 2% of portfolio on any trade
- Monitors positions, trails stops, and learns from closed trades
- Adapts to market regime (bullish/sideways/bearish) — trades bigger in bull markets, smaller in bear
- Auto-disables strategies that stop performing (edge decay tracking)
- Runs unattended via macOS launchd on a daily schedule

**Section 3: Quickstart**
```
1. Clone the repo
2. pip install -r requirements.txt
3. Copy .env.example to .env and fill in Alpaca keys
4. python test_simulation.py          # verify install
5. python orchestrator.py --scan      # see today's signals
6. python orchestrator.py             # run full pipeline (places paper orders)
```

Note: step 3 mentions `.env` but the system actually reads from shell env vars (exported in `~/.zshrc` or `~/.bashrc`). The quickstart should say: "Export the variables listed in `.env.example` to your shell profile (`~/.zshrc` or `~/.bashrc`)" — not imply a dotenv loader. **Dependency:** `.env.example` is created by Plan 05-01 (portability). If Plan 05-01 hasn't run yet, this step should reference the two required vars directly: `ALPACA_API_KEY` and `ALPACA_API_SECRET`.

**Section 4: Architecture (text, not diagram)**
Brief paragraph: orchestrator.py is the brain, scanner.py finds setups, executor.py places bracket orders, reconcile.py validates local state vs broker. Shell wrappers (at_open.sh, monitor.sh, eod.sh) are called by launchd on schedule.

**Section 5: The 5 strategies**
One sentence each, matching CLAUDE.md descriptions.

**Section 6: Configuration**
Point to `config.py` and explain the key knobs: `risk_per_trade_pct`, `max_open_positions`, `use_trailing_stops`. Link to `docs/SETUP-GUIDE.md` for full tuning guide.

**Section 7: Schedule**
The 3-line launchd table from CLAUDE.md (at_open, monitor, eod with times).
Link to `setup_launchd.sh` for installation.

**Section 8: Links**

- `CONTRIBUTING.md` — how to add strategies
- `CONCEPTS.md` — plain-language trading glossary

## Task 2: Write CONTRIBUTING.md

Create `CONTRIBUTING.md` at repo root. Structure:

**Section 1: Adding a New Scanner Strategy**

Walk through the pattern using PULLBACK as the example:
1. Add a module-level function to `scanner.py` following the pattern of `scan_pullback()` (not a class method — there is no Scanner class)
2. The function signature is `def scan_my_strategy(ticker: str, config: AgentConfig) -> Signal | None:` — it fetches its own data via `fetch_data(ticker)`, runs analysis, and returns a `Signal` namedtuple or `None`
3. Register it in the `scanners` list inside `run_full_scan()` — add a tuple like `("My Strategy", scan_my_strategy)` (line ~578)
4. Add any new config parameters to the `AgentConfig` dataclass in `config.py`
5. No registration needed in `edge_tracker.py` — it auto-creates strategy entries on first recorded trade

**Section 2: Adding a Standalone Strategy**

For strategies that don't fit the scanner pattern (like wheel_strategy.py):
1. Create a new file in `strategies/`
2. Follow the pattern of `trailing_ladder.py` — standalone runner with its own config section
3. Wire it into `eod.sh` if it should run on schedule

**Section 3: Code conventions**
- All files use `config.py`'s `AgentConfig` dataclass
- Bracket orders are the default — every buy must have stop-loss and take-profit
- State goes in JSON files (positions.json, etc.) — these are gitignored
- Tests go in `test_*.py` files at repo root
- No API keys in code, ever

**Section 4: Running tests**
```
python test_simulation.py    # Full pipeline, synthetic data
python test_executor.py      # Bracket orders, slippage, reconciliation
```

**Done when:**
- CONTRIBUTING.md exists
- A reader could add a 6th scanner strategy by following the guide
- No personal identifiers
